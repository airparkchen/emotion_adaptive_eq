# 個人化先驗與偏好記憶研究草案

Date: 2026-04-15

## 1. 文件目的

本文件整理目前關於「個人化先驗」的初步討論，目的是：

1. 記錄為什麼現有 tree search 之外，還值得加入個人化記憶
2. 說明這個方向要解決的問題
3. 整理目前較可行的初步方法
4. 作為後續研究與系統升級的規劃草案

本文件討論的內容不是取代現有的 tree-based search，而是思考：

> 如何在現有 coarse-to-fine EQ tree 上加入可累積的使用者偏好，降低冷啟動成本並加速收斂。

## 2. 為什麼值得討論個人化先驗

目前的 closed-loop tree search 有兩個很自然的限制：

1. 若目標是盡量接近使用者的 `Top 1 EQ`，搜尋成本會上升
2. 在 cold start 情境下，系統一開始對使用者完全沒有先驗，只能靠當次試聽結果一步一步判斷

這代表：

1. 需要較多節點測試
2. reward 成本會更高
3. 使用者第一次使用時容易花較長時間才能收斂

因此值得思考的方向是：

> 能不能利用歷史回饋，為使用者建立一個偏好先驗，使系統在進入 tree search 前，就能先對某些分支有較合理的優先排序？

## 3. 這個方向想解決的問題

目前 tree search 已經能做到：

1. 將 preset library 轉成 coarse-to-fine 的搜尋空間
2. 透過情緒 reward 逐步判斷節點好壞
3. 以 reward change 決定 deeper / sibling / fallback

但若進一步考慮長期個人化，還可以想解決以下問題：

1. **冷啟動問題**
   - 新使用者第一次使用時，系統完全沒有偏好先驗
2. **節點測試成本高**
   - 若每次都要從 root 接近盲搜，會消耗較多時間與 reward 預算
3. **跨 session 記憶缺失**
   - 同一位使用者下次再次使用時，若系統完全重來，會浪費已經取得的偏好資訊
4. **相似特徵的泛化能力不足**
   - 若某個 feature 已明顯帶來負回饋，目前系統未必能把這個資訊有效泛化到其他也包含該 feature 的節點

## 4. tree 已經用 feature 分群，為什麼還需要 user vector

這裡需要先釐清一個容易混淆的點：

1. tree 使用 feature，是為了定義搜尋結構
2. user vector 使用 feature，是為了記錄使用者偏好權重

也就是說：

- tree feature 回答的是：
  - 「這個節點代表什麼 EQ 特徵？」
- user vector 回答的是：
  - 「這個使用者對這些 EQ 特徵偏好如何？」

因此兩者雖然都使用 feature，但角色不同：

```text
tree = search structure
user vector = preference prior
```

它們不衝突，反而可以自然結合。

## 5. 初步個人化概念

### 5.1 節點仍然保留 tree 結構

現有系統仍以 EQ tree 作為主要搜尋空間：

1. coarse 層先決定大方向
2. medium / fine 層逐步細化
3. reward 決定 deeper / sibling / fallback

### 5.2 新增個人化偏好記憶

在 tree 結構之外，為每位使用者建立一個偏好向量或偏好權重表，記錄：

1. 對哪些 feature 傾向正回饋
2. 對哪些 feature 傾向負回饋
3. 哪些 feature 目前尚不確定

這個偏好記憶不直接決定答案，而是用來：

1. 排序後續節點的優先級
2. 在 cold start 後的下一輪使用中提供較好的起點
3. 在多個 sibling 間做更有效率的選擇

## 6. 什麼是 feature-level memory

目前 tree 節點本來就已經可被拆成 feature，例如：

1. `stage`
   - coarse / medium / fine
2. `region`
   - low / mid / high
3. `direction`
   - boost / cut / neutral
4. `band group`
   - 例如 `b1_2`, `b3_4`, `b5_6`
5. `magnitude`
   - 例如 `+4`, `-2`, `+1`

因此可以把每個節點表示成一個 feature vector：

```text
node_i = [high_boost, mid_cut, b3_4_plus, coarse, magnitude_4, ...]
```

再把每個使用者表示成一個 preference vector：

```text
user_u = [w_high_boost, w_mid_cut, w_b3_4_plus, ...]
```

之後可用：

```text
predicted_preference(node) = user_vector · node_vector
```

來估計某個節點對該使用者的先驗吸引力。

## 7. 一個具體例子

假設某位使用者在某次探索中：

1. 測到 `high:boost` 分支中的一個 child
2. 該 child 包含 feature `b3_4:+`
3. 使用者給出負向回饋或較差 reward

那系統可以更新：

```text
pref(b3_4:+) -= 某個值
```

之後若在其他分支，例如 `high:cut`，又出現也帶有 `b3_4:+` 的 child：

1. 系統不一定直接排除它
2. 但可以降低它的探索優先級

這樣的好處是：

1. 偏好可以跨分支泛化
2. 不必等到每個節點都實際測過才學到資訊

## 8. 為什麼不建議一開始就直接做 SVD

我們討論過可參考 SVD 或 matrix factorization 的概念，因為它們確實擅長：

1. 使用 `user x item` 偏好矩陣
2. 學 latent preference factor
3. 做冷啟動後的推薦排序

但目前這個專案若直接上標準 SVD，有幾個現實限制：

1. **資料很 sparse**
   - 每位使用者不會測太多節點
2. **reward 很昂貴**
   - 無法快速取得大量 user-item 互動
3. **item 並非純平面列表**
   - 節點本身有 tree 結構與 feature 結構
4. **可解釋性很重要**
   - 目前系統需要能說明為何偏向某個分支

因此較合理的結論是：

> SVD / matrix factorization 可作為後續中長期方向參考，但第一版不建議直接作為主要方法。

## 9. 目前最可行的初步方案

### 9.1 V1：Feature-based user profile

這是目前最推薦的第一版方法。

做法：

1. 為每個節點建立 feature tag / feature vector
2. 為每位使用者建立可更新的 preference vector
3. 每次節點測試完後，依照 reward 對相關 feature 做正負更新
4. 下一次排序 sibling 或 coarse 分支時，將 user preference 納入優先級計算

可以很簡化地寫成：

```text
node_priority = tree_priority + user_preference_bonus
```

其中：

1. `tree_priority`
   - 來自原本 tree search 的結構優先性
2. `user_preference_bonus`
   - 來自該使用者對節點 feature 的歷史偏好

### 9.2 這個方法的優點

1. 與現有 tree feature 相容
2. 可解釋性高
3. 不需要大量資料就能開始累積
4. 可以同時記錄正回饋與負回饋
5. 可用於 cold start 後的快速適應與跨 session 記憶

### 9.3 這個方法的限制

1. feature 設計若太細，資料會非常 sparse
2. 一次負回饋不代表 feature 本身一定不好，可能只是與其他 feature 組合後變差
3. 因此第一版應該：
   - 降低優先級，而不是直接永久排除
   - 先以 coarse / medium feature 為主，不要一開始就過度依賴 fine feature

## 10. 可參考的演算法方向

### 10.1 Content-based recommendation

最適合目前第一版。

做法是：

1. item 使用 feature 表示
2. user 用 feature 偏好表示
3. 根據兩者匹配程度決定節點優先級

對目前系統最相容，也最容易落地。

### 10.2 Contextual bandit

若之後要更正式處理：

1. exploration / exploitation 平衡
2. reward 高成本
3. 不確定性估計

則 contextual bandit 很值得參考。

對應方式可理解為：

1. `arm` = 可測的下一個節點
2. `context` = 使用者 profile + 當前 stage + 當前 path
3. `reward` = emotion-based node reward

這個方向很適合作為 V2 / V3 研究。

### 10.3 Matrix factorization / SVD

較適合作為更後期的方向，前提是：

1. 使用者數量夠多
2. user-item 互動夠多
3. 願意接受較低可解釋性

### 10.4 Bayesian / preference learning

若未來想把 reward 的不確定性、偏好後驗更新正式建模，這也是很值得參考的研究方向。

但第一版不建議直接上，因為實作與資料需求都較重。

## 11. 建議的演進路線

### V1

1. 純 tree-based closed-loop search
2. 無個人記憶
3. 先完成 reward / threshold / policy 校準

### V2

1. 加入 feature-based user preference memory
2. 記錄 coarse / medium feature 的正負回饋
3. 將 preference memory 用於排序 sibling 與 coarse 分支優先級

### V3

1. 引入 contextual bandit 或更正式的 exploration policy
2. 納入多使用者資料形成 shared prior
3. 評估是否值得升級到 latent factor / matrix factorization 類方法

## 12. 目前較適合的研究問題

若要把這條線整理成研究或後續開發議題，目前最值得優先探討的是：

1. tree node 的 feature tag 該如何定義，才能兼顧可解釋性與泛化能力
2. 使用者偏好該累積到 coarse / medium / fine 哪個粒度
3. 正負回饋應如何更新 feature 權重
4. 是否應讓負向 feature 只降低優先級，而不是直接排除
5. preference memory 是否能實際降低 cold start 成本與節點測試數

## 13. 暫時結論

目前最合理的結論是：

1. 個人化先驗值得保留為後續發展方向
2. 第一版不建議直接做 SVD 或完整協同過濾
3. 最適合的初步可行方案是：
   - 使用現有 tree node 的 feature tag
   - 為每位使用者建立可更新的 preference vector
   - 根據正負回饋調整節點探索優先級

一句話總結：

> tree 用 feature 定義搜尋空間；user vector 用 feature 記錄偏好權重。前者是結構，後者是先驗，兩者可以自然結合，形成一個更省測試成本的個人化 EQ 推薦系統。
