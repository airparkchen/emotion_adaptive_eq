# EQ 情緒驅動個人化推薦系統完整提案說明

Date: 2026-04-14

## 1. 提案目的

本文件用來完整整理目前 EQ 自適應推薦系統的提案架構，作為：

1. 自我回顧與整理想法的主文件
2. 與其他人討論專案時的說明基礎
3. 後續驗證實驗與規格細化的上位文件

本文件會說明：

1. 我們到底在解什麼問題
2. 為什麼採用 preset EQ 推薦系統
3. coarse-to-fine tree 的角色與目前做法
4. 演算法目前仍需要補完的關鍵問題
5. 驅動方向與驗證方式應如何思考

## 2. 問題定義

本專案的核心不是在完整 10-band EQ 空間中直接搜尋最佳解，而是：

> 從一組具有代表性的 preset EQ 候選集合中，透過情緒回饋做漸進式個人化推薦，最後收斂到最適合使用者的 EQ。

因此這是一個：

- `preset-based recommendation`
- `emotion-driven closed-loop optimization`
- `coarse-to-fine personalization`

的系統。

## 3. 為什麼採用 preset EQ 候選集合

### 3.1 不把問題定義成完整分類

若理論上將 `low / mid / high` 三個區段各自離散成：

- `boost`
- `neutral`
- `cut`

則可形成 `3 x 3 x 3 = 27` 種組合。

這種分類方式在概念上成立，但目前專案不採用它作為主要產品定義，原因是：

1. 我們沒有足夠依據定義 27 組都具有實際價值且彼此可辨識的 preset。
2. EQ 不只和方向有關，也和幅度有關，因此完整空間會遠大於 27。
3. 即使建立 27 組，也不保證都能形成可感知、可比較、可驗證的使用者選擇。

因此我們目前更合理的定義是：

> 先建立一組高價值且具代表性的 preset 候選集合，再由系統在這個候選集合上進行個人化搜尋。

### 3.2 preset 的來源依據

目前 preset 的建立依據可來自：

1. 文獻支持的曲線，例如 Harman
2. 音樂平台或耳機產品中常見的調音類型
3. 工程與主觀聽感篩選後認為具代表性的 EQ

因此 preset 的角色是：

> 代表性的聲音原型（sound archetypes），而不是理論上完整覆蓋所有偏好空間的最終答案。

### 3.3 對 preset 數量的理解

我們目前不主張「12 個 preset 一定足夠所有使用者」，而是主張：

> 目前的 preset library 已足以作為第一版候選骨架，支撐 tree-based progressive search。

在工程上，目前可以這樣描述：

1. 若 preset 太少，coarse tree 會快速退化成簡單選單
2. 若希望 coarse 層形成多個主要群組且每群仍有子群空間，learnable preset 不宜太少
3. 以目前架構來看：
   - `9` 組 learnable preset 可作為最低可行規模
   - `12` 組 learnable preset 是較合理的第一版規模

## 4. 個人化依據

本系統的個人化依據，不是先驗假設某使用者屬於某種固定 EQ 類型，而是：

> 根據使用者對不同頻段調整方向的情緒回饋差異，逐步推定其偏好方向。

這代表系統會逐步判斷：

- 是否偏好高頻更亮或更柔
- 是否偏好中頻更突出或更後退
- 是否偏好低頻更強或更弱

因此個人化的本質是：

- `feedback-driven preference estimation`

而不是：

- `pre-labeled user classification`

## 5. EQ tree 的角色與切分方法

### 5.1 tree 的角色

tree 的目的不是直接表示完整 preset，而是：

> 將 preset library 轉成一個可逐步探索的搜尋路徑。

也就是：

- preset library 是候選集合
- tree 是搜尋結構

### 5.2 coarse-to-fine 的基本想法

搜尋流程採用：

1. `coarse`
   - 先看大區段差異
2. `medium`
   - 再看較細的區段差異
3. `fine`
   - 最後做單 band 或局部微調

這樣做的理由是：

1. 前期調整要夠有感，讓回饋能有效區分方向
2. 後期再細修，避免一開始就在細節上浪費太多測試成本

### 5.3 目前程式中的 tree 生成

目前實作邏輯是：

1. 先用 `flat` 當 baseline，計算每個 preset 的 delta 向量
2. coarse 層在 `low / mid / high` 等大區段上，找最能區分目前候選集合的 feature
3. coarse 分群不是只看平均值，而是：
   - 取該區段局部曲線
   - 用 L1 距離做 medoid-based clustering
   - 做平衡分配，讓 coarse 群組盡量平均
4. medium / fine 層再用 band pair 與 single band 做局部細分
5. 每個節點的 `delta_curve` 使用該子集在該特徵區段上的中位數代表值

## 6. 為什麼節點 delta 使用中位數

我們討論過另一種做法：使用子集合的 common core（類似最大公因曲線）。

最後目前先保留中位數，原因是：

1. 若子集合內差異太大，common core 很容易縮得太小
2. 前段 coarse / medium 搜尋需要有感調整
3. 系統在單一候選後本來就會進入 full preset refine，因此前段不需過度保守

因此目前採用：

> 節點 delta 用子集合代表值（中位數）來做 coarse-to-fine 搜尋，最後再做 full preset 貼合。

## 7. 演算法目前最核心但尚未完全定稿的部分

目前真正還沒完全 specification 化的，是 `emotion-to-decision` 這段。

我們可以拆成三個規格：

### 7.1 Evaluation

問題是：

> 某個 EQ 節點在一段時間內的情緒表現，要怎麼算？

需要定義：

1. emotion model 輸出格式
2. observation window
3. 每個節點至少需要多少資料
4. reward 要用平均、投票、加權還是其他方式

### 7.2 Decision

問題是：

> 這個節點比上一個好嗎？

需要定義：

1. 如何比較 reward 差值
2. 何時算 `improve`
3. 何時算 `stable`
4. 何時算 `worsen`

### 7.3 Search Policy

問題是：

> 知道 improve / stable / worsen 之後，tree 要怎麼走？

需要定義：

1. coarse 層是 breadth-first 還是 depth-first
2. improve 時往哪個 child
3. stable 時要不要試 sibling
4. worsen 時要退到哪一層
5. 什麼時候可以停止

## 8. 為什麼 reward 成本是現在最重要的產品限制

這一點是最近討論中最重要的新結論：

> 每一次 reward 的取得，都需要讓使用者實際聽一段時間。

因此 reward 是高成本訊號，而不是可以無限取得的資訊。

這代表：

1. tree search 不能被設計成完整展開搜尋
2. 演算法必須考慮：
   - `max_total_search_time`
   - `max_tested_nodes`
   - `minimum_effective_gain`
   - `good_enough_reward`
3. 產品版系統的目標不一定是理論最優，而是：
   - 用有限預算找到夠好的解

## 9. 目前的 V1 與下一步方向

### 9.1 V1 的基本想法

目前草案中的 V1 是：

1. 每個 EQ 節點對應一個固定 observation window
2. 用該 window 內的平均 emotion score 當 reward
3. 比較前後 reward 差值，轉成：
   - `improve`
   - `stable`
   - `worsen`
4. coarse 層偏 breadth-first
5. medium / fine 偏 depth-first
6. 候選收斂到單一 preset 後，進入 full preset refine

### 9.2 V1 的限制

目前這版 V1 還沒有完全把 reward 高成本納入核心設計，所以它更像是：

- 一版能跑通 closed-loop 的規格草案

而不是：

- 最終產品版的 budget-aware search

### 9.3 V1.5 的方向

若要更接近產品可用版本，下一步建議加入：

1. `budget-aware search`
2. `early stop`
3. `minimum effective gain`
4. `good enough reward`
5. coarse 層的 top-k 分支保留策略

## 10. 目前已形成的初步共識

在最近的討論中，我們已經把 `emotion -> reward -> decision` 這條鏈條先收斂出一版可討論的初始規格。

### 10.1 Emotion output -> Reward

目前的共識是：

1. 不以 binary 輸出作為主要訊號
2. 先使用區分力較明顯的 `emotion_score` 連續值
3. reward 定義需要同時考慮：
   - 同一 EQ 下的自然波動（noise）
   - EQ 切換後第一筆資料較不穩的轉換效應

因此目前暫時採用：

```text
node_reward = 0.3 * window_1 + 0.7 * window_2
```

也就是：

1. 每個節點至少收 2 個 windows
2. 第一筆保留，但降權
3. 第二筆較接近穩定狀態，因此給較高權重

這個 weighting 不是定稿，而是一個根據現有小規模資料提出的工程化初值，後續可用更多資料調整。

### 10.2 Reward -> EQ adjust direction

目前的共識是：

1. 不直接用絕對 emotion score 區間判斷 EQ 好壞
2. 先使用 reward 的相對變化：

```text
delta_reward = reward_t - reward_(t-1)
```

3. direction 判定必須考慮：
   - 同一條件下的 noise
   - 主觀偏好帶來的最小有效差異
   - 不同 stage 的變化量級

因此目前初始 threshold 可先從 `8` 開始，理由是：

1. 它大致接近目前可觀察到的 noise floor 上方
2. 又不會大到完全吞掉目前資料中的較小偏好差異

但這裡也已經有一個明確的新結論：

```text
T_coarse > T_medium > T_fine
```

也就是 coarse / medium / fine 很可能需要不同 threshold，而不是共用同一個門檻。

### 10.3 stable 的意義不只一種

目前我們也已經注意到，`stable` 並不一定都代表相同意思。

至少存在兩種可能：

1. `neutral stable`
   - 沒有明顯變好，也沒有明顯變差
   - 通常表示此方向資訊量不足，可改試 sibling

2. `good stable`
   - current node 本身已經在高 reward 區域
   - child 沒有大幅更好，但不代表此方向沒價值
   - 這種情況下可能仍值得做 limited sibling comparison，以避免只停在「還不錯」而不是更接近 `Top 1`

因此後續 decision rule 不應只看單一 threshold，還可能需要同時考慮：

1. `delta_reward`
2. `current_reward`
3. `stage`

### 10.4 下一步：用資料來校準規格

目前 reward 與 threshold 都已有初始方向，但還不適合直接定稿。

因此下一步的重點，不是再抽象討論規則，而是設計一個資料蒐集與離線 replay 的探索實驗，去校準：

1. observation window 長度
2. reward weighting
3. threshold 初值
4. coarse / medium / fine 是否需要不同 threshold
5. `stable` 在高 reward 區域是否應採不同 decision 邏輯

## 11. 驅動方向如何說明

目前比較合理的說法是：

> emotion model 不直接輸出 EQ 調整方向，而是輸出某個節點當前的 reward signal；真正的調整方向是由 reward、tree structure、以及 search policy 共同決定。

這樣可避免把情緒模型誤解成直接的 left / right classifier。

## 11. 驗證方法如何設計

### 11.1 第一版驗證：Top-N 命中

可先做：

1. 固定音樂
2. 讓受測者先聽所有 preset
3. 做主觀排序
4. 取 Top 2 / Top 3
5. 再跑 adaptive search
6. 驗證最終結果是否落在 Top-N 中

### 11.2 補充指標

除了 Top-N hit rate 外，也建議記錄：

1. `tree node distance`
2. `preset rank distance`
3. `convergence time`
4. `convergence steps`

### 11.3 探索性內部實驗

在正式驗證前，建議先做小規模內部實驗，目的不是驗證整個系統，而是先估計：

1. emotion output 的穩定性
2. 合理的 window 長度
3. threshold 的候選範圍
4. 使用者能容忍的總測試時間

## 12. 模型不確定性需要被納入設計

目前應避免假設：

> emotion model 可以 100% 準確對應主觀排序。

比較合理的假設是：

1. 模型有資訊量，但有噪音
2. 模型適合當 reward signal
3. 模型不適合直接當一次就決定答案的 classifier

因此後續 decision rule 應考慮：

1. stable 區間
2. threshold
3. smoothing / 投票 / averaging
4. fallback 與 hysteresis

## 13. 總結

目前整個提案可用一句話總結：

> 我們希望建立一個以 preset EQ 候選集合為基礎、以 coarse-to-fine tree 為搜尋結構、以情緒回饋為 reward signal 的個人化推薦系統，並在有限測試成本下，逐步收斂到使用者偏好的 EQ。

目前真正的重點已不再只是「tree 要怎麼建」，而是：

1. 怎麼定義 emotion evaluation
2. 怎麼把 reward 轉成 decision
3. 怎麼在高成本 reward 條件下設計合理的 search policy
4. 怎麼用驗證與探索實驗把這些 spec 定下來
