# 驗證與探索實驗規劃

Date: 2026-04-14

## 1. 文件目的

本文件整理目前可規劃的兩類實驗，但這兩類實驗的角色不同，不能混在一起解讀：

1. 參數校準 / 資料蒐集實驗
   - 目的是了解資料特性
   - 幫助我們決定 window、threshold、reward weighting、stopping、budget 等規格
2. 正式驗證實驗
   - 目的是驗證完整演算法是否有效
   - 需在 reward、threshold、policy 較穩定後再進行

因此目前的優先順序應為：

1. 先做資料蒐集與參數校準
2. 等演算法規格較穩後，再做正式 closed-loop 驗證

補充說明：

本階段的資料蒐集實驗，主要不是要直接證明整個 EQ 推薦演算法已經有效，而是要先建立：

1. `noise` 的量級理解
2. reward 的量級分布
3. 正向 / 中性 / 負向回饋的大致區間
4. `window / weighting / threshold` 的候選設定

這些結果將優先用於：

1. 建構 V1 版本的 `Evaluation`
2. 建構 V1 版本的 `Decision`
3. 建構 V1 版本的 `Search Policy`

同時，這批資料也可作為後續個人化與 `Contextual Bandit` 導入時的初始全域先驗（global prior）基礎。

## 2. 參數校準 / 資料蒐集實驗

### 2.1 定位

這類實驗的目的不是直接證明演算法已經成功，而是先回答：

1. emotion output 是否穩定可分
2. reward 應怎麼定義
3. threshold 應落在哪個區間
4. coarse / medium / fine 是否需要不同 decision 規則

也就是說，這類實驗比較像：

- calibration study
- exploratory study

而不是最終 validation。

更具體地說，這一階段希望收集到的資料，至少能支撐：

1. 單一 EQ / 節點下的情緒波動 `noise`
2. reward 的正向、負向與中性變化量級
3. 不同 stage 的 reward 變化差異
4. 不同 `window / weighting / threshold` 組合的離線比較
5. 後續 global prior 或個人化偏好初始化的資料來源

### 2.2 目前較合理的初步流程

目前討論下來，一個可行的資料蒐集版本是：

1. 受測者先聽完整個 preset library
2. 由受測者做主觀排序，記錄 `Top1 / Top2`
3. 測全部 coarse 節點
4. 再測 `Top1 preset` 所屬分支下的全部節點與 preset
5. 每次播放固定 4 個 windows
   - 以目前估計約為 1 分鐘
6. 全程記錄 `emotion_score`

這樣的資料可同時支援：

1. 正面 / 中性 / 負面的 reward delta 分布估計
2. coarse 層的 reward 變化觀察
3. medium / fine 層的 reward 變化觀察
4. window、weighting、threshold 的離線調參

### 2.3 這類實驗的解讀原則

這類實驗不應以「有沒有直接命中 Top1」作為唯一判準，因為：

1. coarse 層本來就只是在抓較粗粒度的偏好方向
2. coarse 未必直接對應最終 Top1 preset
3. 情緒模型與主觀偏好之間本來就可能存在不完全一致

因此在這類資料蒐集實驗中，更合理的觀察包括：

1. coarse 最佳分支是否命中 `Top1` 所屬分支
2. coarse 最佳分支是否落在 `Top1 / Top2` 所屬範圍
3. 正確分支與最佳分支的 reward gap 是否很小
4. 不同 stage 下的 reward 量級是否明顯不同

換句話說，這類實驗主要是用來：

- 觀察 tree 特徵是否有辨識力
- 幫助我們調參

而不是直接作為最終有效性的唯一證據。

## 3. 正式驗證實驗：Top-N 命中驗證

### 3.1 核心問題

我們希望回答：

> 演算法最後收斂到的 EQ，是否接近使用者主觀偏好的高分候選？

### 3.2 初版流程

1. 固定音樂內容
2. 讓受測者聽所有 preset EQ
3. 由受測者做主觀評分或排序
4. 取出 `Top N`（例如 Top 2 / Top 3）
5. 執行一輪 adaptive EQ search
6. 觀察最終結果是否落在使用者的 Top N 中

這類驗證與前面的資料蒐集不同，這裡應該讓受測者實際跑完整個 adaptive search，直到：

1. 收斂到 `stable`
2. 或達到最大步數 / 最大時間

也就是它必須更接近真實 closed-loop 使用情境。

### 3.3 可記錄指標

1. `Top-N hit rate`
   - 最終結果是否落在 Top N
2. `Top rank distance`
   - 最終結果距離 Top 1 差幾名
3. `tree node distance`
   - 最終結果距離使用者真實偏好節點差幾個節點
4. `convergence steps`
   - 幾步內到 stable EQ
5. `convergence time`
   - 幾秒內到 stable EQ

### 3.4 補充建議

Top-N 驗證法是合理的第一版，但建議補兩點：

1. 加 baseline 對照
   - `flat`
   - random preset
   - static best-known preset
2. 重複 ranking 或 pairwise check
   - 降低受測者單次排序的偶然誤差

## 4. 正式驗證實驗：節點距離誤差

### 4.1 為什麼值得做

只看 Top-N 命中有時太粗，因為：

- 演算法可能沒命中 Top 1
- 但其實只差一個節點或一層局部 refine

因此可以額外記錄：

> 最終收斂節點距離使用者偏好節點有多遠。

### 4.2 可定義的距離

1. `tree path distance`
   - 兩個節點在 tree 上的最短路徑長度
2. `preset rank distance`
   - 最終結果與 Top 1 的排名差距
3. `EQ vector distance`
   - 最終 EQ 與使用者偏好 preset 的向量差距

這一類指標有助於看出：

- 系統是否已經接近正確區域
- 即使沒完全命中，是否仍有實際價值

## 5. 參數校準實驗：Window 與 Threshold

### 5.1 核心問題

我們需要知道：

1. emotion model 的有效 window 多長
2. reward 變化是否穩定可分
3. threshold 大概應該設在哪個範圍

補充背景：

- 既有經驗顯示，情緒模型與主觀偏好之間雖然有對應性，但不應假設模型可 100% 穩定還原使用者排序。
- 因此 threshold、window、投票或 smoothing 的設計，不只是為了數學穩定，也是為了吸收模型本身的不確定性。

### 5.2 可行的內部探索實驗

可先做個人或小規模內部測試：

1. 固定音樂
2. 聽完整個 preset library
3. 每個 preset 保持固定時間
4. 同步記錄情緒模型輸出

目的：

1. 觀察不同 preset 是否真的能拉開 emotion score
2. 觀察同一個 preset 在不同時間窗下的穩定性
3. 估計可接受的 observation window
4. 初步估計 improve / stable / worsen threshold

目前已有的先驗觀察：

1. 既有 E1(flat) 與 E2(主觀 Top1) 小規模資料顯示，`emotion_score` 在多位受測者上通常能觀察到 E2 高於 E1。
2. binary 輸出區分力偏弱，不足以單獨作為 reward。
3. EQ 剛切換後的第一個 window 常較不穩，推測需要降權或排除。
4. 因此探索實驗的重點不只是「有沒有差異」，還包括：
   - 差異是否穩定
   - 第一筆是否需要降權
   - 兩個 windows 是否已足夠形成可用 reward

### 5.3 這種實驗可能看出的東西

1. 某些 preset 間分數差異很小
   - 代表 tree 分裂可能過細
2. 某些 preset 需要較長時間才看出差異
   - 代表 evaluation window 不能太短
3. 同一 preset 的情緒輸出波動很大
   - 代表 decision rule 需要 smoothing 或投票
4. 某些人對所有 preset 差異都不明顯
   - 代表演算法對該類使用者可能需要更保守策略
5. 模型對主觀偏好的對應度有限
   - 代表 decision rule 不應依賴單次結果
   - 需要 trend、stable 區間、以及容錯設計
6. 第一筆轉換效應明顯
   - 代表 reward 定義需考慮 window weighting
   - 也代表 threshold 不應只看單筆差值

### 5.4 目前可先採用的 reward / threshold 初值

依據目前 `doc/EQ_exp_data/` 中的 E1(flat) / E2(主觀 Top1) 小規模資料，可先得到一版探索實驗起始值：

1. `reward` 先使用連續值 `emotion_score`
2. 每個 EQ 節點先收集 2 個 windows
3. 使用加權聚合：

```text
node_reward = 0.3 * window_1 + 0.7 * window_2
```

4. `delta_reward = reward_t - reward_(t-1)`
5. 初始 threshold 先用：

```text
delta_reward >= +8 -> improve
delta_reward <= -8 -> worsen
otherwise -> stable
```

這個初值的依據是：

1. 目前資料中，E1 與 E2 的平均差距約從 `+6.9` 到 `+28.6`
2. 同一 EQ 內的分數波動約落在 `7` 到 `11` 的量級
3. 因此 threshold 應至少接近 noise floor，但不能大到吞掉較小的有效差異

這裡的數字不是定稿，而是：

> 一組供探索實驗與小規模內部測試使用的起始參數。

後續更新方式建議固定為：

1. 持續收集更多 E1 / E2 或其他對照條件資料
2. 估計同一 EQ 下的自然波動範圍
3. 估計明顯偏好差異的典型 reward 差距
4. 重新調整 weighting 與 threshold

也就是說，後續不是直接改數字，而是依照：

- `noise floor`
- `偏好差距量級`
- `第一筆轉換效應是否明顯`

這三個依據來更新 reward / threshold 設定。

### 5.5 探索實驗應額外驗證的兩個問題

根據目前討論，除了估計單一初始 threshold 之外，還需要額外驗證兩件事：

1. `stage-specific threshold` 是否必要
2. 高 reward 區域中的 stable 是否應與一般 stable 區分

原因：

1. coarse 節點的變化通常較大，fine 節點的變化通常較小
2. 若全部共用同一個 threshold，fine 階段可能因改善幅度較小而過度被判成 `stable`
3. 若系統目標是更接近 `Top 1`，則高 reward 區域內的小差異可能仍有探索價值

因此後續探索實驗除了記錄整體命中率，也建議記錄：

1. coarse / medium / fine 各 stage 的 reward 變化量級
2. 各 stage 在不同 threshold 下的 `improve / stable / worsen` 比例
3. 高 reward 區域中，sibling comparison 是否能提高 Top-N 命中

這些觀察可用來決定：

1. 是否從單一 threshold 升級成 `T_coarse / T_medium / T_fine`
2. 是否需要把 stable 分成：
   - 一般 stable
   - 高分 stable
3. 在高分 stable 情況下，是否保留 limited sibling check

### 5.6 後續資料蒐集與離線調參方向

若後續要專門蒐集資料來校準 window 與 threshold，建議將目標定成：

1. 建立每個 stage 下的 reward 變化分布
2. 建立同一 EQ 下的 noise floor
3. 建立高偏好 EQ 與普通 EQ 的典型差距
4. 用離線 replay 比較不同參數組合

可優先調整的參數包括：

1. window 長度
   - 例如 10 / 15 / 20 秒
2. reward weighting
   - 例如 `0.5/0.5`、`0.3/0.7`
3. threshold
   - 例如 `5 / 8 / 10 / 12`
4. stage-specific threshold
   - 例如 coarse 較大、fine 較小

離線 replay 可比較的指標包括：

1. Top-N hit rate
2. tree node distance
3. convergence time
4. fallback 次數
5. sibling 測試次數
6. 是否容易過早停止在「還不錯」而不是較接近 Top 1 的節點

## 6. 參數校準實驗：Reward 成本與時間預算

### 6.1 核心問題

如果 reward 很貴，我們要知道：

1. 使用者最多能接受多久的搜尋時間
2. 每輪最多可接受幾個節點測試
3. 什麼情況下應該 early stop

### 6.2 可規劃的內部實驗

1. 設定不同 observation window
   - 例如 10 秒 / 15 秒 / 20 秒
2. 模擬不同最大步數
   - 例如 4 / 6 / 8 steps
3. 比較：
   - 命中率
   - 收斂時間
   - 受測者疲勞感

目的：

- 找到「可接受使用時間」與「有效搜尋能力」的平衡點

## 7. 建議的短期實驗順序

### 7.1 第一階段：參數校準 / 資料蒐集實驗

目的：

- 先了解 emotion output 特性
- 粗估 window 與 threshold

建議：

1. 小規模內部測試
2. 固定音樂
3. 先做主觀 Top1 / Top2 ranking
4. 測全部 coarse 節點
5. 測 Top1 所屬分支的全部節點與 preset
6. 每段收 4 個 windows
7. 記錄 emotion output 與主觀評分

### 7.2 第二階段：正式驗證實驗

目的：

- 驗證 adaptive algorithm 是否能找到使用者喜歡的候選

建議：

1. 先做 preset ranking
2. 讓受測者實際跑完整個 adaptive search，直到 stable
3. 計算 Top-N hit rate、node distance、convergence time
4. 加 baseline 對照
