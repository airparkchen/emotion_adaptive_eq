# 演算法實作前需釐清的議題整理

> Note
> 本文件已由 `doc/EQ架構/議題整理.md` 接手持續維護。
> 此檔案保留作為歷史版本參考，後續請以 `doc/EQ架構/` 內文件為主。

Date: 2026-04-14

## 1. 文件目的

本文件用來整理目前 EQ 自適應演算法在真正落地前，必須先釐清的問題。  
目的不是直接給最終答案，而是建立一個清楚的思考脈絡，說明為什麼這些議題需要被討論，以及它們彼此之間的依賴關係。

## 2. 為什麼現在要討論這些事情

目前專案已經具備：

1. 一組可用的 preset library
2. 一個可從 preset 自動建立 tree 的方法
3. 一個 coarse-to-fine 的 EQ optimizer
4. 一個可模擬 emotion input 的 prototype loop

但這還不足以定義真正的產品演算法，因為目前最關鍵的閉環邏輯仍未完全被 specification 化：

- 如何評估某一個 EQ 節點的情緒表現
- 如何比較兩個節點的好壞
- 如何根據這個比較結果決定 tree 應該前進、換路、撤回，還是維持

換句話說，目前我們已經有 `preset`、`tree`、`optimizer`，但 `emotion-to-decision` 的規格還沒有真正定清楚。

## 3. 目前需要切開思考的三個 spec

### 3.1 Evaluation Spec

定義：

> 某一個 EQ 節點在一段時間內，情緒表現要怎麼被量化。

這裡會牽涉到：

1. emotion model 的原始輸出是什麼
   - binary label
   - continuous score
   - probability / confidence
2. 模型最短需要多長的輸入 window 才能做出可靠預測
3. 每個 EQ 節點需要觀察多久，才能完成一次 evaluation
4. 一次 evaluation 要使用幾筆 emotion output

這一塊很重要，因為若 evaluation 定義不清楚，後面的 decision 就沒有穩定基礎。

### 3.2 Decision Spec

定義：

> 當前 EQ 節點的情緒表現，和上一個節點相比，是變好、差不多，還是變差。

這裡會牽涉到：

1. 如何計算情緒變化
   - 單筆比較
   - 平均值比較
   - 投票
   - 加權平均
   - 差分趨勢
2. trend 的輸出型態
   - `improve`
   - `stable`
   - `worsen`
3. threshold 如何設定
4. 是否需要 smoothing / hysteresis / confidence filter

這一塊的核心是：emotion model 不直接告訴你 EQ 要往哪走，它只能先提供一個 reward signal，再由系統把 reward 轉成 trend。

### 3.3 Search Policy Spec

定義：

> 知道目前 trend 之後，tree 該怎麼走。

這裡會牽涉到：

1. coarse 層應該先廣度搜尋還是先深度搜尋
2. medium / fine 層是否應該改用不同策略
3. `improve` 時要前進到哪個 child
4. `stable` 時要換 sibling、停留、還是再觀察一次
5. `worsen` 時要回 parent、回上一個最佳節點，還是回 root
6. 什麼時候算 stable EQ，可以停止搜尋

這一塊是整個 closed-loop 的核心控制器，也是目前最需要討論清楚的部分。

## 4. 這三個 spec 的依賴關係

這三個問題不能同時亂討論，順序很重要。

比較合理的順序是：

1. 先定 `Search Policy`
2. 再定 `Evaluation Spec`
3. 再定 `Decision Spec`
4. 最後補 `Stopping Criteria`

原因是：

- 若不知道 tree 怎麼跑，就無法知道一個節點至少要嘗試多久
- 若不知道每個節點要觀察多久，就無法定義 emotion input 的最低需求
- 若沒有 evaluation 的定義，就無法穩定定義 improve / stable / worsen

因此目前最適合的討論方式不是直接問「emotion 要怎麼算」，而是先問：

> 我們預期 tree 在 coarse / medium / fine 三個階段，應該怎麼探索？

## 5. 目前最需要關注的主題

### 5.1 Emotion model 的介面限制

目前有一個非常現實的限制：

> emotion model 能多快輸出一筆有效結果，不是演算法可以任意決定，而是取決於模型本身。

例如若 PPG-based 模型需要：

- 至少 15 秒資料
- 才能從 PPG 算出 HRV
- 再由 HRV 推估情緒

那 EQ 決策頻率就不應設計得快於這個有效 window。

因此我們需要先釐清：

1. emotion model 最短有效推論窗口多長
2. 更新頻率是否可以比窗口更快
3. 是否能接受 overlapping windows

### 5.2 EQ 節點要如何被評估

目前很自然會想到一個問題：

> 如果節點 A 的情緒結果是正面，那接下來應該往左還是往右？

這其實說明了：

- emotion model 不應直接輸出「左 / 右」
- 它應該先輸出「這個節點目前好不好」

也就是：

1. emotion model 提供 reward
2. 系統比較 reward 變化
3. policy 再把 reward 轉成 traversal action

### 5.3 如何定義 stable EQ

產品上不能一直調整，因此一定要回答：

1. 什麼情況下算 stable
2. stable 是 temporary stable 還是 final stable
3. 到葉節點後若情緒變差，是否允許 fallback

這一塊會直接影響使用者體驗。

## 6. 目前可以先接受的規劃原則

在還沒把細節完全定下來之前，我們至少可以先接受這幾個原則：

1. emotion model 提供的是 `reward signal`，不是直接的 EQ 方向
2. traversal direction 必須由 `reward + tree structure + policy` 共同決定
3. coarse / medium / fine 很可能需要不同搜尋策略
4. EQ 的停止條件必須被定義，不能無限制持續調整
5. 實驗設計要能驗證：
   - 是否找到使用者偏好的候選
   - 是否收斂得夠快
   - 是否比 baseline 更有效

## 7. 下一步建議

目前最合理的下一步是：

1. 先寫出 `Emotion 回饋與搜尋策略規格草案`
2. 用一版簡化但可討論的 V1 policy 作為起點
3. 後續再逐步細化：
   - emotion input spec
   - trend decision rule
   - fallback rule
   - stopping criteria

## 8. 總結

目前的核心不是再增加更多 preset 或重寫 tree，而是把整個 closed-loop 中最關鍵的三件事 specification 化：

1. 怎麼評估一個 EQ 節點
2. 怎麼比較兩個節點
3. 怎麼根據比較結果決定樹的 traversal

只要這三塊定清楚，後續不論是實驗設計、產品包裝、還是演算法迭代都會容易很多。
